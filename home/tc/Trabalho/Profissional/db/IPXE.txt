    Criar uma máquina virtual no QEMU com suporte à inicialização PXE, utilizando iPXE para conectar a um servidor Micro Core Linux, Dnsmasq e Busybox httpd, é um processo avançado que envolve a configuração de vários componentes. Vamos detalhar cada etapa:

## Visão Geral do Fluxo

1.  **Máquina Virtual QEMU:** Será configurada para iniciar a partir da rede (PXE).
2.  **iPXE:** Embarcado na VM, ele substituirá o firmware PXE tradicional e fornecerá mais flexibilidade.
3.  **Servidor TFTP (Dnsmasq):** Fornecerá o arquivo de boot do iPXE.
4.  **Servidor DHCP (Dnsmasq):** Atribuirá um endereço IP à VM e informará o servidor TFTP e o arquivo de boot.
5.  **Servidor HTTP (Busybox httpd):** Servirá o kernel do Micro Core Linux e o initrd.
6.  **Micro Core Linux:** O sistema operacional leve que será carregado e executado.

## Pré-requisitos

*   **Linux Host:** O sistema operacional onde você executará o QEMU.
*   **QEMU:** Instalado em seu host.
*   **iPXE:** Você precisará compilar ou baixar um binário do iPXE pré-compilado. Geralmente, um binário EFI (como `ipxe.efi`) é mais fácil de integrar com o QEMU moderno.
*   **Dnsmasq:** Instalado em seu host.
*   **Busybox:** Instalado em seu host e configurado para usar o httpd.
*   **Micro Core Linux:** O arquivo de imagem do kernel (`vmlinuz`) e o ramdisk (`initrd.gz`) do Micro Core Linux. Você pode baixá-los do site oficial do Tiny Core Linux.

## Etapas Detalhadas

### Passo 1: Preparar os Arquivos do Micro Core Linux

1.  Crie um diretório para os arquivos de boot na sua máquina host. Por exemplo:
    ```bash
    mkdir -p ~/pxe_boot/microcore
    cd ~/pxe_boot/microcore
    ```
2.  Baixe os arquivos do Micro Core Linux. Você precisará do kernel (`vmlinuz`) e do initrd (`initrd.gz`). Você pode encontrar esses arquivos nos downloads do Tiny Core Linux. Por exemplo, para a versão 13.1:
    ```bash
    wget https://tinycorelinux.net/13.1/x86_64/release/vmlinuz64
    wget https://tinycorelinux.net/13.1/x86_64/release/initrd64.img
    # Renomeie para facilitar
    mv vmlinuz64 vmlinuz
    mv initrd64.img initrd.gz
    ```

### Passo 2: Configurar o Dnsmasq (Servidor DHCP e TFTP)

1.  Crie um arquivo de configuração para o Dnsmasq. Por exemplo, `~/pxe_boot/dnsmasq.conf`:
    ```
    # Configuração do Dnsmasq para PXE boot
    interface=br0 # Se você estiver usando uma bridge, caso contrário, use a interface de rede do seu host
    listen-address=127.0.0.1,192.168.1.100 # Adapte ao IP do seu host
    bind-interfaces

    # DHCP configuration
    dhcp-range=192.168.1.200,192.168.1.250,12h # Faixa de IPs para as VMs
    dhcp-option=option:router,192.168.1.1 # Gateway padrão (geralmente o IP do seu host ou roteador)
    dhcp-option=option:dns-server,8.8.8.8 # Servidor DNS (opcional)

    # PXE boot configuration
    enable-tftp
    tftp-root=/home/seu_usuario/pxe_boot # Caminho para os arquivos de boot (importante!)
    pxe-service=tag:0x82,FFFF,ipxe.efi # Solicita o ipxe.efi para placas de rede com tag 0x82
    # Se você não usar tag, pode usar algo como:
    # pxe-service=tag:0,67,"ipxe.efi"
    # pxelinux.cfg/default é comum para syslinux, mas para iPXE precisamos apontar para o script do iPXE
    # O iPXE irá carregar o boot.ipxe do TFTP.

    # Se você quiser que o iPXE baixe o boot.ipxe automaticamente do TFTP (sem necessidade de pxe-service)
    # dpkg-reconfigure dnsmasq
    # O prompt vai perguntar sobre tftp. Responda sim.
    # Ou manualmente:
    # dhcp-boot=tag:0x82,ipxe.efi # Define o boot file para o iPXE.
    # Mas para iPXE, o ideal é que ele carregue um script.

    # Vamos focar em um setup onde o iPXE busca um script de boot.
    # Vamos configurar o dnsmasq para servir o ipxe.efi
    # E o ipxe.efi vai buscar o boot.ipxe do TFTP.
    ```

    **Explicação das opções:**

    *   `interface`: A interface de rede do seu host que o Dnsmasq irá escutar. Se você estiver usando um bridge, use o nome da bridge (ex: `br0`). Caso contrário, use a interface de rede principal do seu host (ex: `eth0`, `enp3s0`).
    *   `listen-address`: O IP do seu host.
    *   `dhcp-range`: A faixa de endereços IP que o Dnsmasq atribuirá às VMs.
    *   `dhcp-option=option:router`: O gateway padrão para as VMs.
    *   `enable-tftp`: Habilita o servidor TFTP no Dnsmasq.
    *   `tftp-root`: O diretório raiz do TFTP. **Este deve ser o diretório onde você colocará o `ipxe.efi` e outros arquivos de boot.**
    *   `pxe-service`: Diz ao Dnsmasq para oferecer um arquivo de boot PXE. `tag:0x82` é frequentemente usado por placas de rede virtuais no QEMU para indicar que elas querem PXE boot. `FFFF` é um timeout. `ipxe.efi` é o nome do arquivo a ser baixado via TFTP.

2.  **Prepare o diretório raiz do TFTP:**
    ```bash
    mkdir -p ~/pxe_boot/tftp_root
    ```

3.  **Copie o `ipxe.efi` para o diretório raiz do TFTP:**
    Você precisará compilar o iPXE ou baixar um binário. Para compilar:
    ```bash
    # No diretório do iPXE
    make bin/ipxe.efi
    cp bin/ipxe.efi ~/pxe_boot/tftp_root/
    ```
    Se você baixou um binário pré-compilado (procure por "ipxe.efi download"), copie-o para `~/pxe_boot/tftp_root/`.

4.  **Crie o script de boot do iPXE (`boot.ipxe`):**
    Este script instruirá o iPXE sobre como inicializar o Micro Core Linux. Crie um arquivo chamado `boot.ipxe` em `~/pxe_boot/tftp_root/`.
    ```bash
    # ~/pxe_boot/tftp_root/boot.ipxe
    #!ipxe
    # Configuração do iPXE para Micro Core Linux

    # Defina o IP do seu servidor HTTP (onde o Busybox estará rodando)
    # Use o IP do seu host que as VMs poderão acessar.
    set server_ip 192.168.1.100 # Adapte ao IP do seu host

    # Defina o caminho para os arquivos do Micro Core Linux no servidor HTTP
    # O caminho deve ser relativo à raiz do servidor HTTP
    set microcore_path /microcore

    # Boot do Micro Core Linux
    kernel http://${server_ip}/microcore/vmlinuz quiet
    initrd http://${server_ip}/microcore/initrd.gz
    # Use o IP do servidor DHCP (dnsmasq) para instruir o kernel
    # Isso é importante para o Micro Core Linux saber de onde obter o rootfs (se necessário)
    # Mas neste caso, o initrd.gz já contém tudo que precisa para iniciar
    # boot proto=dhcp dhcp-client-identifier=ia-pd # Exemplo de boot proto
    # Se o Micro Core Linux for usar DHCP para montar o rootfs, ajuste aqui
    # O Micro Core Linux tende a carregar tudo no initrd.
    boot
    ```

    **Importante:**
    *   Substitua `192.168.1.100` pelo endereço IP do seu host Linux que rodará o Dnsmasq e o Busybox httpd.
    *   Certifique-se de que o `microcore` no `set microcore_path` corresponda à estrutura de diretórios que você criará no servidor HTTP.

5.  **Inicie o Dnsmasq:**
    ```bash
    sudo dnsmasq -C ~/pxe_boot/dnsmasq.conf
    ```
    Se você tiver um Dnsmasq rodando como serviço, pare-o antes (`sudo systemctl stop dnsmasq`) e execute este comando para usar sua configuração específica. Monitore os logs do Dnsmasq para verificar se há erros.

### Passo 3: Configurar o Busybox httpd (Servidor HTTP)

1.  Crie um diretório para servir os arquivos do Micro Core Linux via HTTP:
    ```bash
    mkdir -p ~/pxe_boot/http_root/microcore
    ```
2.  Copie os arquivos do Micro Core Linux para este diretório:
    ```bash
    cp ~/pxe_boot/microcore/vmlinuz ~/pxe_boot/http_root/microcore/
    cp ~/pxe_boot/microcore/initrd.gz ~/pxe_boot/http_root/microcore/
    ```
3.  Execute o Busybox httpd. Você pode fazer isso em um terminal separado.
    ```bash
    cd ~/pxe_boot/http_root
    busybox httpd -f -p 80 -h .
    ```
    *   `-f`: Executa em foreground.
    *   `-p 80`: Escuta na porta 80.
    *   `-h .`: Define o diretório raiz do servidor HTTP como o diretório atual (`~/pxe_boot/http_root`).

    **Importante:** Certifique-se de que o IP do seu host (usado no `boot.ipxe`) esteja acessível na porta 80. Se houver um firewall, você pode precisar abrir a porta 80.

### Passo 4: Criar a Máquina Virtual QEMU

Agora você criará a máquina virtual QEMU e a configurará para inicializar via PXE.

```bash
qemu-system-x86_64 \\
    -machine q35,accel=kvm \\
    -cpu host \\
    -m 2048 \\
    -smp 2 \\
    -device virtio-net-pci,netdev=net0 \\
    -netdev user,id=net0 \\
    -drive if=pflash,format=raw,readonly=on,file=/usr/share/ovmf/OVMF_CODE.fd \\
    -drive if=pflash,format=raw,file=/tmp/OVMF_VARS.fd \\
    -device ich9-usb-uhci1 -device usb-ehci \\
    -device usb-tablet \\
    -vga virtio \\
    -display sdl,gl=on \\
    -cdrom /dev/null \\
    -boot order=d
```

**Explicação das opções:**

*   `-machine q35,accel=kvm`: Usa a arquitetura de máquina `q35` (mais moderna) e aceleração KVM para melhor desempenho.
*   `-cpu host`: Usa as mesmas instruções da CPU do seu host.
*   `-m 2048`: Aloca 2048 MB de RAM para a VM.
*   `-smp 2`: Aloca 2 núcleos de CPU para a VM.
*   `-device virtio-net-pci,netdev=net0`: Adiciona um adaptador de rede `virtio-net` à VM.
*   `-netdev user,id=net0`: Configura a rede `user` para a VM. Isso cria um NAT simples, onde a VM pode acessar a rede do host, mas o host não inicia conexões para a VM. Para uma inicialização PXE mais robusta, **você pode precisar usar `netdev=tap,id=net0,script=no,downscript=no` e configurar uma bridge de rede (`br0`) no seu host**. No entanto, para um teste rápido, `user` pode funcionar se o Dnsmasq estiver configurado para escutar na interface correta.
*   `-drive if=pflash,format=raw,readonly=on,file=/usr/share/ovmf/OVMF_CODE.fd`: Monta o firmware UEFI de código aberto (OVMF). **Você precisa ter o pacote `ovmf` instalado no seu host Linux.**
*   `-drive if=pflash,format=raw,file=/tmp/OVMF_VARS.fd`: Monta o arquivo de variáveis UEFI. **Você precisará criar este arquivo primeiro:**
    ```bash
    cp /usr/share/ovmf/OVMF_VARS.fd /tmp/OVMF_VARS.fd
    ```
    **Importante:** O QEMU precisa que o iPXE seja carregado como um dispositivo de boot EFI.
*   `-device ich9-usb-uhci1 -device usb-ehci -device usb-tablet`: Configurações USB.
*   `-vga virtio`: Usa um adaptador gráfico `virtio`.
*   `-display sdl,gl=on`: Usa a saída gráfica SDL com aceleração OpenGL.
*   `-cdrom /dev/null`: Evita que o QEMU tente inicializar de um CD-ROM.
*   `-boot order=d`: Define a ordem de boot para dispositivos que suportam boot (neste caso, queremos que ele tente iniciar pela rede).

**Configurando o Boot EFI para usar iPXE:**

A parte crucial para fazer o iPXE funcionar com QEMU e OVMF é instruir o firmware EFI a carregar o `ipxe.efi` como o primeiro dispositivo de boot de rede.

1.  **Inicie a VM com a configuração inicial:**
    Execute o comando QEMU acima. A VM deve iniciar e tentar inicializar pela rede. Se tudo estiver configurado corretamente, ela obterá um IP do Dnsmasq e baixará o `ipxe.efi` via TFTP.

2.  **Entre no Boot Manager do OVMF:**
    Quando a VM inicia e o `ipxe.efi` é baixado, você verá a interface do iPXE. O script `boot.ipxe` deve ser executado automaticamente, tentando baixar o kernel e o initrd via HTTP. Se isso funcionar, o Micro Core Linux iniciará.

    Se não iniciar automaticamente ou se você quiser configurar o boot de forma mais persistente, você precisará entrar no menu de configuração do OVMF (Boot Manager). Você pode fazer isso de algumas maneiras:
    *   **Botão "e" na tela de boot do OVMF:** Algumas versões do OVMF exibem um menu onde você pode pressionar 'e' para editar as opções de boot.
    *   **Entrando no "Device Manager" do OVMF:** Navegue pelas opções do OVMF para encontrar as configurações de boot.

3.  **Configure a Ordem de Boot no OVMF:**
    Dentro do Boot Manager do OVMF, você precisará configurar a ordem de boot para priorizar a inicialização de rede, e especificar o protocolo (EFI network boot).
    *   Procure por opções como "Boot Order" ou "Boot Manager".
    *   Adicione uma entrada para "Network Boot" ou "UEFI Network".
    *   Se o QEMU/OVMF oferecer a opção de especificar o arquivo de boot, você pode tentar apontar diretamente para `ipxe.efi` ou um caminho relativo. No entanto, o iPXE configurado no Dnsmasq geralmente é suficiente.

    **O objetivo é fazer com que o firmware EFI tente inicializar via rede.** Quando ele faz isso, ele solicita um arquivo de boot via PXE. O Dnsmasq responde com o `ipxe.efi`.

### Dicas e Solução de Problemas

*   **Rede:** A configuração de rede é o ponto mais crítico. Se a VM não obtiver um IP, verifique:
    *   Se o Dnsmasq está rodando e configurado corretamente.
    *   Se a interface de rede especificada no `dnsmasq.conf` é a correta.
    *   Se o QEMU está configurado para usar uma rede que o Dnsmasq possa ver. Usar uma bridge de rede (`br0`) no host é geralmente a abordagem mais confiável para PXE boot em VMs.
    *   O firewall no seu host Linux.
*   **TFTP:** Se o `ipxe.efi` não for baixado, verifique:
    *   Se o `tftp-root` no `dnsmasq.conf` aponta para o diretório correto onde `ipxe.efi` está.
    *   Se o Dnsmasq tem permissão para ler esse diretório.
*   **iPXE Script:** Se o `ipxe.efi` for baixado, mas o script `boot.ipxe` não carregar o kernel/initrd:
    *   Verifique se o caminho para `boot.ipxe` está correto no diretório `tftp-root`.
    *   Verifique se o `server_ip` e o `microcore_path` no `boot.ipxe` estão corretos.
    *   Verifique se o Busybox httpd está rodando e servindo os arquivos nas portas corretas.
    *   Verifique se a VM pode acessar o servidor HTTP (usando `ping` dentro da VM se ela iniciar um shell).
*   **OVMF e EFI Boot:** Certifique-se de que o pacote `ovmf` esteja instalado e que os caminhos para os arquivos `.fd` estejam corretos. A configuração da ordem de boot no OVMF é essencial.
*   **Micro Core Linux:** Você pode precisar ajustar os parâmetros do kernel passados na linha de comando do `boot.ipxe` para o Micro Core Linux, dependendo de como você deseja que ele inicie (por exemplo, para montar um rootfs de rede). O `initrd.gz` do Tiny Core Linux geralmente é autônomo e carrega tudo na memória.

### Exemplo de Configuração de Bridge de Rede (Mais Robusta)

Para uma configuração de rede mais robusta, onde o Dnsmasq e o servidor HTTP são facilmente acessíveis pela VM, é recomendado configurar uma bridge de rede no seu host Linux.

1.  **Instale `bridge-utils`:**
    ```bash
    sudo apt install bridge-utils # Debian/Ubuntu
    sudo yum install bridge-utils # CentOS/Fedora
    ```

2.  **Crie uma bridge (ex: `br0`):**
    Você pode configurar isso usando `/etc/network/interfaces` (Debian/Ubuntu) ou `nmcli` (Fedora/CentOS).

    **Exemplo com `/etc/network/interfaces` (Debian/Ubuntu):**
    ```
    auto eth0 # Sua interface de rede física
    iface eth0 inet manual

    auto br0
    iface br0 inet static
        address 192.168.1.100 # IP do seu host
        netmask 255.255.255.0
        gateway 192.168.1.1
        dns-nameservers 8.8.8.8
        bridge_ports eth0
        bridge_stp off
        bridge_fd 0
        bridge_maxwait 0
    ```
    Reinicie a rede: `sudo systemctl restart networking`

3.  **Atualize `dnsmasq.conf`:**
    Mude `interface=br0` e `listen-address=192.168.1.100` (o IP da bridge).

4.  **Atualize o comando QEMU:**
    Use a configuração de rede da bridge:
    ```bash
    qemu-system-x86_64 \\
        ...
        -device virtio-net-pci,netdev=net0 \\
        -netdev tap,id=net0,ifname=tap0,script=no,downscript=no \\
        ...
    ```
    E então conecte o `tap0` à bridge manualmente:
    ```bash
    sudo brctl addif br0 tap0
    sudo ip link set tap0 up
    ```
    Você pode automatizar isso com scripts.

Este é um setup complexo, mas seguindo estes passos metodicamente, você conseguirá criar sua máquina virtual com inicialização PXE através do iPXE. Lembre-se de adaptar os caminhos e endereços IP à sua configuração específica."
